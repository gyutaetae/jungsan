# 아이디어톤 심사위원 관점 최종 리뷰

## 한 줄 결론

`정산`은 "간편장부 엑셀 생성기"로 발표하면 약하다. 하지만 "소상공인이 홈택스 신고 전에 가장 오래 걸리는 영수증 정리, 비용 분류, 증빙 확인, 입력값 준비를 자동화하는 신고 준비 작업대"로 발표하면 충분히 경쟁력 있다.

우승을 노리려면 메시지를 이렇게 잡아야 한다.

> 사진첩 속 영수증과 카드 거래내역이, 검토 가능한 홈택스 입력값으로 바뀝니다.

현재 MVP가 홈택스 자동 제출까지 하지 않는 것은 약점이 아니다. 세금 도메인에서는 자동 제출보다 `검토 가능한 준비 자동화`가 먼저다. 인증, 보안, 법적 책임, 세무 판단 리스크를 무리하게 넘지 않고도 사용자의 시간을 줄이는 지점이 명확하기 때문이다.

## 심사위원 평가

### 현재 점수 감각

- 문제 공감도: 높음
- MVP 현실성: 높음
- 데모 설득력: 중상
- 차별성 전달력: 보완 필요
- 우승 가능성: 메시지와 시연 순서만 잘 잡으면 있음

가장 조심해야 할 오해는 이것이다.

> 결국 엑셀만 만들어주는 것 아닌가요?

이 질문이 나오면 발표가 흔들린다. 그래서 산출물을 "엑셀"이라고 먼저 말하면 안 된다. 엑셀은 전달 형식일 뿐이고, 실제 가치는 `신고 화면에 옮길 값`, `공식 서식 기반 초안`, `확인된 거래만 반영되는 신뢰 구조`다.

## 프로젝트가 잘 나타나고 있는가

현재 `README.md`, `PRD`, `idea-thon-review.md`의 방향은 프로젝트를 꽤 잘 나타낸다. 특히 다음 메시지는 좋다.

> 정산은 홈택스를 대체하지 않습니다. 세무사를 대체하지도 않습니다. 대신 소상공인이 신고 전에 가장 오래 붙잡히는 영수증 정리, 비용 분류, 장부 초안 작성, 홈택스 입력값 준비를 줄입니다.

다만 발표 자료와 제품 설명에서는 아직 `신고 준비`보다 `간편장부/엑셀`이 먼저 보일 위험이 있다. 심사위원은 짧은 시간 안에 "그래서 사용자가 무엇을 덜 하게 되는가?"를 봐야 한다.

따라서 제품을 설명하는 순서는 다음이 더 강하다.

1. 영수증과 거래내역이 흩어져 있다.
2. 사용자는 홈택스 신고 전에 같은 값을 여러 번 옮겨 적는다.
3. `정산`은 AI로 거래를 읽고, 사용자가 확인한 값만 반영한다.
4. 확인된 값이 필요경비, 소득금액, 증빙별 합계, 신고 준비 문서 초안으로 바뀐다.
5. 사용자는 홈택스에 입력할 숫자를 바로 확인한다.

## 와우 포인트

### 메인 와우 포인트

> 사진첩 속 영수증이 홈택스 입력값으로 바뀐다.

이 문장이 가장 강하다. 기술적으로는 OCR, 엑셀 파싱, 장부화, 요약, 파일 생성이지만 심사위원에게는 이렇게 보여야 한다.

> 사용자가 영수증 사진을 올리면, AI가 거래 카드로 정리하고, 확인된 값만 홈택스 입력용 요약과 신고 준비 문서 초안으로 바꿉니다.

### 보조 와우 포인트

- AI가 세금을 대신 판단하지 않고, 모르는 값은 확인 필요로 남긴다.
- 모든 거래는 사용자가 `확인`해야 완료된다.
- 확인된 거래만 필요경비와 소득금액 계산에 반영된다.
- 간편장부는 최종 결과물이 아니라 홈택스 입력값을 만들기 위한 기반 데이터다.
- 원본 파일을 저장하지 않는 구조는 초기 MVP에서 신뢰 포인트가 된다.

### 데모에서 보여줘야 할 순간

심사위원이 "아, 이거 그냥 OCR이 아니네"라고 느끼는 순간은 업로드가 아니라 요약 화면이다.

데모의 클라이맥스는 다음 순서여야 한다.

1. 샘플 영수증 또는 거래내역 업로드
2. 거래 카드 생성
3. 금액, 계정과목, 증빙유형 확인
4. `확인` 버튼 클릭
5. 오른쪽 요약 패널에서 필요경비와 예상 소득금액이 바뀌는 장면
6. `홈택스 입력용 요약`과 `신고 준비 문서` 다운로드

발표자는 이때 이렇게 말하면 된다.

> 단순히 글자를 읽는 것이 아니라, 신고 전에 필요한 숫자로 정리해주는 것이 핵심입니다.

## 공식 맥락

국세청 간편장부 안내에 따르면 간편장부는 거래가 발생한 날짜 순서대로 수입, 비용, 거래처, 거래내용, 부가세, 증빙 등을 기록하는 구조다. 비용 및 매입거래는 거래 건별 기재가 기본이고, 증빙은 세금계산서, 계산서, 카드, 현금영수증, 영수증 등으로 구분된다.

따라서 `정산`이 영수증과 카드 거래내역을 거래일, 거래처, 금액, 공급가액, 부가세, 계정과목, 증빙유형으로 바꾸는 것은 단순 표 정리가 아니다. 신고 준비에 필요한 장부 구조로 바꾸는 작업이다.

## 지금 잘한 점

### 1. 문제 정의가 현실적이다

소상공인의 진짜 병목은 마지막 제출 버튼이 아니라 그 전 단계다.

- 영수증은 사진첩에 흩어져 있다.
- 카드 거래내역은 엑셀로 따로 있다.
- 홈택스는 입력값을 요구한다.
- 장부와 신고 서식은 또 다른 구조를 가진다.

이 문제는 실제 사용자가 반복적으로 겪는 문제라 공감도가 높다.

### 2. MVP 범위를 잘 잘랐다

홈택스 자동 제출까지 가면 인증, 보안, 법적 책임, 세무 판단 문제가 생긴다. 아이디어톤 MVP에서 그 영역을 무리하게 구현하면 오히려 위험한 서비스처럼 보일 수 있다.

현재 범위는 더 설득력 있다.

- AI가 초안을 만든다.
- 사용자가 확인한다.
- 확인된 값만 요약과 문서에 반영한다.
- 산출물은 제출본이 아니라 검토용 초안이다.

### 3. 데모가 끝까지 이어진다

업로드에서 끝나는 서비스가 아니라, 검토 카드, 필요경비 요약, 홈택스 입력용 요약, 신고 준비 문서 다운로드까지 이어진다. 이 흐름은 단순 OCR 앱보다 강하다.

### 4. 신뢰 설계가 있다

세금 서비스에서 "AI가 알아서 다 합니다"는 오히려 불안하다. `정산`은 AI가 틀릴 수 있다는 전제를 제품 구조에 반영한다.

- 모르는 값은 억지로 채우지 않는다.
- 사용자가 확인한 거래만 완료 처리한다.
- 확인 필요 항목을 따로 보여준다.

이것은 약점이 아니라 세금 도메인에 맞는 설계다.

## 지금 부족한 점

### 1. 홈택스 입력값이 더 전면에 나와야 한다

현재 제품은 신고 준비까지 가지만, 발표자가 설명을 잘못하면 "장부 엑셀 다운로드 서비스"처럼 보일 수 있다.

개선 방향:

- 요약 패널 제목을 `홈택스 입력 준비`로 더 강하게 잡는다.
- 총수입금액, 필요경비, 예상 소득금액을 "신고 화면에 옮길 값"으로 설명한다.
- 각 값 옆에 `복사` 버튼이 있으면 더 좋다.
- "확인된 거래만 반영" 문구를 화면에 보여준다.

### 2. 공식 서식 대응감이 약하다

HWP 원본을 직접 채우지 않는 것은 MVP에서 합리적이다. 하지만 사용자가 기대한 "공식 양식에 딱 맞춘다"는 느낌은 약해질 수 있다.

개선 방향:

- 거래 카드나 요약 화면에 `공식 서식 반영 위치`를 보여준다.
- 예: `간편장부 > 비용 > 소모품비`
- 예: `총수입금액 및 필요경비명세서 > 필요경비 합계`
- 예: `간편장부소득금액계산서 > 소득금액`

실제 HWP 편집 없이도 "공식 구조에 맞춰 정리된다"는 인상을 줄 수 있다.

### 3. 매출 데이터 자동화가 약하다

현재는 비용 증빙 정리에 강하고, 총수입금액은 직접 입력에 가깝다. 실제 신고에서는 매출도 중요하다.

발표에서는 이렇게 방어한다.

> MVP에서는 가장 빈번하고 귀찮은 비용 증빙 정리에 집중했습니다. 다음 단계에서는 카드매출, 플랫폼 정산서, 계좌 입금 내역 업로드를 붙여 총수입금액까지 자동화할 계획입니다.

### 4. 발표 자료가 프로젝트와 다르면 치명적이다

기존 PDF 구성안이 시니어 디지털 비서(CUEE) 중심이라면, 현재 `정산` 프로젝트 발표에는 사용하면 안 된다. 심사위원은 발표 자료와 데모가 다르면 팀의 문제 정의가 흔들린다고 판단한다.

`정산` 전용 발표 자료는 반드시 새로 잡아야 한다.

## 우승을 위한 발표 구조

### 1. 오프닝

> 소상공인에게 세금 신고가 어려운 이유는 마지막 제출 버튼 때문만이 아닙니다. 진짜 문제는 그 전에 있습니다. 영수증은 사진첩에 있고, 카드 내역은 엑셀에 있고, 홈택스는 또 다른 입력값을 요구합니다.

### 2. 문제 정의

소상공인은 신고 전 자료 정리에서 시간을 잃는다.

- 영수증 찾기
- 카드내역 정리
- 거래처와 금액 확인
- 증빙유형 구분
- 필요경비 합계 계산
- 홈택스 입력값 확인

### 3. 솔루션

> `정산`은 영수증 사진과 거래내역을 검토 가능한 거래 카드로 바꾸고, 사용자가 확인한 값만 홈택스 입력용 요약과 신고 준비 문서 초안으로 만듭니다.

### 4. 데모

데모는 샘플 데이터로 먼저 완주해야 한다. OCR/API는 발표장에서 실패할 수 있으므로 보너스 시연으로 두는 편이 안전하다.

추천 순서:

1. 샘플 데이터 불러오기
2. 거래 카드 확인
3. 하나의 거래를 수정하고 `확인`
4. 필요경비 요약 변화 확인
5. 홈택스 입력용 요약 확인
6. 신고 준비 문서 다운로드
7. 시간이 남으면 영수증 이미지 인식 시연

### 5. 차별점

> 정산은 OCR 앱이 아닙니다. 카드사 엑셀 뷰어도 아닙니다. 흩어진 거래 자료를 신고 전 입력값으로 바꾸는 작업대입니다.

차별점은 세 가지로 정리한다.

- OCR 이후의 장부화
- AI 결과를 사람이 확인하는 신뢰 구조
- 홈택스 입력값과 공식 서식 초안까지 이어지는 흐름

### 6. 클로징

> 우리는 세금 신고를 무리하게 완전 자동화하지 않습니다. 대신 사용자가 믿고 확인할 수 있는 방식으로, 신고 전 가장 오래 걸리는 정리 시간을 줄입니다.

## 발표 슬라이드 추천 구성

1. 표지: `정산 - 사진첩 속 영수증을 홈택스 입력값으로`
2. 문제: 신고가 어려운 이유는 제출 전 자료 정리다
3. 사용자 상황: 사진첩, 카드 엑셀, 홈택스 화면, 장부 양식이 흩어져 있음
4. 솔루션: 영수증/거래내역 → 거래 카드 → 확인 → 입력값 요약
5. 데모 흐름: 업로드부터 신고 준비 문서까지
6. 와우 포인트: 사진 속 영수증이 홈택스 입력값으로 바뀌는 장면
7. 신뢰 설계: AI 초안 + 사용자 확인 + 확인된 값만 반영
8. 공식 맥락: 간편장부/필요경비/증빙유형 구조
9. 경쟁 서비스 비교: OCR 앱, 카드사 엑셀, 세무대행과의 차이
10. 시장성: 개인사업자/프리랜서/1인 창업자 중심
11. BM: 무료 체험 + 월 구독 + 세무사/기관 제휴
12. 로드맵: 비용 증빙 → 매출자료 → 홈택스 입력 가이드 → 세무사 검토 연동
13. 마무리: `정산은 신고 버튼이 아니라, 신고 전 시간을 줄입니다`

## 예상 질문과 답변

### Q. 결국 엑셀만 만들어주는 것 아닌가요?

아닙니다. 엑셀은 전달 형식입니다. 핵심은 영수증과 거래내역을 장부 항목으로 구조화하고, 계정과목, 증빙유형, 공급가액, 부가세, 필요경비, 예상 소득금액을 신고 전에 확인 가능한 값으로 바꿔주는 것입니다.

### Q. 카드사 엑셀 다운로드와 뭐가 다른가요?

카드사 엑셀은 원천 데이터입니다. `정산`은 그 데이터를 신고 준비에 필요한 장부 구조로 바꾸고, 사용자가 확인한 값만 요약과 신고 준비 문서에 반영합니다.

### Q. 왜 홈택스 자동 제출까지 안 하나요?

자동 제출은 인증, 보안, 법적 책임, 세무 판단 리스크가 큽니다. MVP에서는 사용자가 실제로 먼저 막히는 신고 전 자료 정리와 입력값 준비를 자동화하는 것이 더 현실적이고 안전합니다.

### Q. 공식 HWP 양식을 그대로 채우나요?

MVP에서는 HWP를 직접 편집하지 않습니다. 웹 환경에서 HWP 직접 조작은 안정성과 호환성 리스크가 큽니다. 대신 공식 항목 구조를 참고한 xlsx 초안과 홈택스 입력용 요약을 생성합니다.

### Q. AI가 틀리면 어떻게 하나요?

AI 결과를 자동 확정하지 않습니다. 모든 거래는 검토 카드에 올라오고, 사용자가 확인 버튼을 눌러야 완료됩니다. 모르는 값은 확인 필요로 남기는 것이 원칙입니다.

### Q. 세무 판단까지 해주나요?

MVP에서는 복잡한 세무 판단을 자동화하지 않습니다. 세무 판단이 필요한 항목은 확인 필요로 남기고, 사용자가 검토할 수 있는 입력값 준비에 집중합니다. 이것이 세금 도메인에 맞는 신뢰 전략입니다.

## 당장 고쳐야 할 우선순위

### 1순위: 발표 자료를 `정산` 전용으로 교체

시니어 디지털 비서/CUEE 관련 슬라이드는 현재 프로젝트와 맞지 않는다. 반드시 소상공인 세금 신고 준비 문제로 다시 구성해야 한다.

### 2순위: 제품 화면에서 `홈택스 입력 준비`를 더 크게 보여주기

장부 다운로드보다 홈택스에 옮길 값을 먼저 보여줘야 한다. 심사위원은 "그래서 신고에 어떻게 도움이 되는데?"를 바로 봐야 한다.

### 3순위: 공식 서식 대응표 추가

기능 구현 난도 대비 발표 효과가 크다. 실제 HWP를 생성하지 않아도 공식 구조에 맞춰 정리된다는 느낌을 줄 수 있다.

### 4순위: 데모 안정화

발표장에서는 샘플 데이터로 전체 흐름을 먼저 완주한다. OCR은 네트워크/API 문제로 실패할 수 있으니 후순위로 둔다.

## 최종 피칭 문장

> 정산은 간편장부 생성기가 아닙니다. 소상공인을 위한 홈택스 신고 준비 작업대입니다. 사진첩 속 영수증과 거래내역을 AI가 먼저 정리하고, 사용자가 확인한 값만 필요경비와 홈택스 입력용 요약, 신고 준비 문서 초안으로 바꿉니다.

## 심사위원 관점 최종 조언

이 프로젝트는 "완전 자동 세금 신고"라고 밀면 위험하다. 아직 홈택스 제출도 없고, 세무 판단 자동화도 제한적이기 때문이다.

대신 "신고 전 자료 정리 자동화"로 잡으면 강하다. 문제는 현실적이고, MVP는 안전하며, 데모 흐름도 명확하다. 우승을 위해서는 기술을 더 크게 말하기보다 사용자의 시간을 얼마나 줄이는지 보여줘야 한다.

핵심은 하나다.

> 정산은 신고를 대신하는 서비스가 아니라, 신고 전 가장 귀찮은 준비 시간을 줄이는 서비스다.
