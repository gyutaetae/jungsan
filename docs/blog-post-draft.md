# 블로그 포스트 초안: 데이터베이스 수업 프로젝트 `정산`

## 추천 제목

영수증 사진과 거래내역을 신고 준비 DB로 바꾸는 서비스, `정산`

## 본문 초안

### 1. 프로젝트를 시작한 이유

이번 데이터베이스 수업 프로젝트에서 우리 팀은 소상공인과 프리랜서가 세금 신고 전에 겪는 자료 정리 문제에 집중했다.

세금 신고가 어려운 이유는 마지막 제출 버튼 때문만은 아니다. 실제로 더 번거로운 과정은 그 전에 있다. 영수증은 사진첩에 흩어져 있고, 카드 거래내역은 엑셀 파일로 따로 존재한다. 홈택스나 장부 양식에 입력하려면 거래일, 거래처, 금액, 공급가액, 부가세, 계정과목, 증빙유형 같은 정보를 다시 정리해야 한다.

우리는 이 반복 작업을 줄이기 위해 `정산`이라는 웹앱을 만들었다.

`정산`은 영수증 이미지와 거래내역 파일을 하나의 장부 데이터로 통합하고, 사용자가 검토한 값을 Supabase Postgres에 저장해 필요경비 요약과 신고 준비 문서 초안을 만들어주는 서비스다.

### 2. 핵심 아이디어

`정산`의 핵심은 AI가 세금 신고를 대신하는 것이 아니다. AI는 영수증에서 거래 정보를 추출하는 보조 역할을 하고, 최종 확인은 사용자가 한다.

우리가 중요하게 생각한 원칙은 다음과 같다.

- AI 결과를 바로 확정하지 않는다.
- 사용자가 검토한 거래만 신뢰 가능한 데이터로 본다.
- 공급가액이나 부가세가 불명확하면 임의로 계산하지 않는다.
- 영수증 이미지와 엑셀 거래내역을 같은 장부 구조로 통합한다.
- 장부 행과 신고 준비 요약을 DB에 저장하고 다시 불러올 수 있게 한다.

이 구조를 통해 사용자는 흩어진 자료를 하나씩 다시 입력하는 대신, 검토 테이블에서 필요한 값만 확인하고 수정할 수 있다.

### 3. 사용자 흐름

서비스 흐름은 다음과 같다.

1. 사용자가 영수증 사진 또는 거래내역 엑셀 파일을 업로드한다.
2. 앱이 입력 데이터를 장부 행으로 변환한다.
3. 사용자는 검토 테이블에서 거래일, 거래처, 금액, 계정과목, 증빙유형을 확인한다.
4. 확인된 거래를 기준으로 필요경비 요약이 계산된다.
5. 총수입금액을 입력하면 예상 소득금액을 확인할 수 있다.
6. 사용자는 현재 장부를 Supabase DB에 저장할 수 있다.
7. 저장된 장부와 총수입금액을 다시 불러올 수 있다.
8. 간편장부 엑셀과 신고 준비 문서 초안을 다운로드한다.

이 흐름에서 가장 중요한 화면은 검토 테이블이다. 검토 테이블은 AI나 엑셀 파싱 결과를 사용자가 확인 가능한 장부 데이터로 바꾸는 지점이다.

### 4. 데이터베이스 설계

이번 프로젝트에서는 Supabase Postgres를 사용했다. Supabase를 선택한 이유는 Postgres 기반이라 데이터베이스 수업에서 중요한 정규화, 외래키, 제약조건, 인덱스, view, RLS를 실제로 설명하기 좋기 때문이다.

현재 DB 설계의 주요 테이블은 다음과 같다.

- `profiles`: 사용자 프로필
- `businesses`: 사용자의 사업장 또는 신고 단위
- `import_batches`: 한 번의 영수증 촬영 묶음 또는 엑셀 업로드 묶음
- `source_files`: 업로드 파일 메타데이터
- `ledger_entries`: 장부 거래 행
- `tax_prep_summaries`: 신고 준비 요약

이 중 가장 중요한 테이블은 `ledger_entries`이다. 영수증 AI 결과와 엑셀 거래내역이 모두 이 테이블의 같은 구조로 저장된다.

### 5. ERD 구조

전체 관계는 다음과 같이 설계했다.

```txt
auth.users 1 ── 1 profiles
profiles 1 ── N businesses
businesses 1 ── N import_batches
import_batches 1 ── N source_files
businesses 1 ── N ledger_entries
import_batches 1 ── N ledger_entries
source_files 1 ── N ledger_entries
businesses 1 ── N tax_prep_summaries
```

한 사용자는 여러 사업장이나 신고 단위를 가질 수 있다. 하나의 사업장에는 여러 업로드 묶음과 장부 거래가 연결된다. 업로드 파일은 장부 거래의 출처를 추적하기 위한 메타데이터로 관리하고, 신고 준비 요약은 장부 거래를 집계한 결과로 저장한다.

### 6. 핵심 테이블: ledger_entries

`ledger_entries`는 검토 테이블의 DB 버전이다. 주요 컬럼은 다음과 같다.

- `business_id`: 어떤 사업장의 거래인지
- `source`: 영수증, 엑셀, 카메라, 샘플 중 어디서 온 데이터인지
- `status`: 대기, 처리 중, 검토 필요, 확인 완료, 실패 상태
- `transaction_date`: 거래일
- `vendor_name`: 거래처명
- `business_number`: 사업자등록번호
- `description`: 거래 내용
- `total_amount`: 총액
- `supply_amount`: 공급가액
- `vat_amount`: 부가세
- `vat_status`: 부가세 상태
- `category`: 계정과목
- `proof_type`: 증빙유형
- `confidence`: AI 추출 신뢰도
- `memo`: 사용자 메모

이 테이블을 기준으로 필요경비 합계, 계정과목별 합계, 증빙유형별 합계, 신고 준비 요약을 만들 수 있다.

### 7. 정규화 관점

처음에는 모든 정보를 클라이언트의 `LedgerEntry` 타입 하나로 다룰 수 있다. 하지만 DB에서는 역할별로 테이블을 나누는 것이 더 적절하다.

우리 프로젝트에서는 사용자, 사업장, 업로드 묶음, 파일 메타데이터, 장부 행, 신고 요약을 분리했다.

이렇게 분리한 이유는 다음과 같다.

- 사용자와 사업장 정보를 장부 거래마다 반복 저장하지 않아도 된다.
- 한 번의 업로드에서 여러 파일과 여러 거래가 생성되는 관계를 표현할 수 있다.
- 원본 파일 저장 여부와 장부 데이터 저장을 분리할 수 있다.
- 신고 준비 요약을 장부 데이터와 별도 스냅샷으로 관리할 수 있다.

즉, 단순히 표 하나에 데이터를 넣는 것이 아니라 실제 업무 흐름에 맞춰 데이터를 분리했다.

### 8. 무결성 규칙

세금 신고 준비 데이터는 정확성이 중요하다. 그래서 DB에도 여러 제약조건을 넣었다.

- `total_amount`는 0 이상이어야 한다.
- `supply_amount`, `vat_amount`도 값이 있다면 0 이상이어야 한다.
- `confidence`는 0 이상 1 이하만 허용한다.
- `vat_status = confirmed`이면 `supply_amount`와 `vat_amount`가 반드시 있어야 한다.
- `tax_year`는 2000년부터 2100년 사이만 허용한다.
- 같은 사용자, 같은 사업장명, 같은 신고연도는 중복 생성할 수 없다.
- 사업장과 신고연도 기준 신고 요약은 하나만 유지한다.

특히 부가세를 임의로 역산하지 않는 점이 중요하다. AI가 모르는 값을 그럴듯하게 채우는 것보다, 모르는 값은 확인 필요 상태로 남기는 것이 세금 도메인에서는 더 안전하다고 판단했다.

### 9. Enum과 상태 관리

Postgres enum을 사용해 상태와 분류값을 제한했다.

- `ledger_source`: `receipt`, `spreadsheet`, `camera`, `sample`
- `ledger_status`: `queued`, `processing`, `needs_review`, `confirmed`, `failed`
- `vat_status`: `confirmed`, `missing`, `estimated`
- `expense_category`: 소모품비, 여비교통비, 접대비 등
- `proof_type`: 카드영수증, 현금영수증, 세금계산서 등

문자열을 자유롭게 저장하면 오타나 잘못된 상태가 들어갈 수 있다. enum을 사용하면 DB 수준에서 허용 가능한 값만 저장되도록 제한할 수 있다.

### 10. 인덱스와 집계 View

사용자가 자주 조회할 데이터에 맞춰 인덱스를 설계했다.

- 날짜순 장부 조회
- 계정과목별 필요경비 집계
- 증빙유형별 집계
- 검토 필요 항목 조회
- 사용자별 신고연도 조회

또한 다음 view를 만들었다.

- `monthly_expense_summary`: 월별 지출 합계
- `category_expense_summary`: 계정과목별 필요경비 합계
- `proof_type_expense_summary`: 증빙유형별 합계
- `needs_review_entries`: 확인 필요 항목

장부 데이터를 단순 저장하는 데서 끝나지 않고, 신고 준비에 필요한 요약을 DB에서 바로 확인할 수 있도록 했다.

### 11. 보안 설계

세무 데이터는 민감하다. 그래서 Supabase RLS를 적용해 사용자별 접근을 분리할 수 있도록 설계했다.

기본 정책은 단순하다.

- 사용자는 자기 프로필만 접근할 수 있다.
- 사용자는 자기 사업장만 접근할 수 있다.
- 장부 행, 업로드 묶음, 파일 메타데이터, 신고 요약은 연결된 사업장의 소유자만 접근할 수 있다.

현재 데모 API는 서버에서 service role key를 사용해 DB에 접근한다. 실제 서비스로 확장하면 Supabase Auth와 RLS 정책을 함께 사용해 사용자별 장부 접근을 더 명확하게 분리할 수 있다.

### 12. 구현한 기능

현재 MVP에서 구현한 주요 기능은 다음과 같다.

- 영수증 이미지 업로드
- 카메라 기반 여러 장 촬영 흐름
- AI 기반 영수증 정보 추출
- 거래내역 `.xlsx`/`.csv` 업로드
- 장부 검토 테이블
- 거래 상태 관리
- 필요경비 요약 계산
- 총수입금액 입력
- 예상 소득금액 계산
- Supabase DB 상태 확인
- 장부 데이터 DB 저장
- 장부 데이터 DB 불러오기
- 간편장부 엑셀 다운로드
- 신고 준비 문서 초안 다운로드

발표 데모에서는 샘플 데이터를 사용해 전체 흐름을 먼저 보여주고, DB 저장과 불러오기로 데이터가 유지되는 장면을 보여주는 것이 좋다.

### 13. 프로젝트를 하며 배운 점

이번 프로젝트를 통해 데이터베이스 설계는 단순히 테이블을 만드는 일이 아니라는 것을 배웠다. 현실의 업무 흐름을 데이터 구조로 바꾸고, 어떤 값을 확정할 수 있고 어떤 값은 검토 상태로 남겨야 하는지 정하는 과정이 중요했다.

또한 AI를 사용하더라도 데이터베이스 관점의 검증과 무결성은 여전히 필요하다. AI가 추출한 값은 초안일 뿐이고, 사용자가 확인한 데이터만 신뢰 가능한 장부 데이터로 다루어야 한다.

특히 Postgres의 enum, check constraint, foreign key, index, view, RLS를 적용하면서 데이터 품질과 보안을 애플리케이션 코드에만 맡기지 않는 설계가 중요하다는 점을 느꼈다.

### 14. 한계와 개선 방향

현재 프로젝트는 수업 발표와 MVP 데모에 맞춰 데모 사용자 중심으로 DB 저장을 구현했다. 실제 서비스로 확장하려면 다음 기능이 필요하다.

- 실제 사용자 로그인
- 사업장별 장부 관리
- 원본 영수증 이미지 선택 저장
- 업로드 이력 상세 화면
- 기간별 거래 검색
- 세무사 검토 연동
- 확인된 거래만 신고 요약에 반영하는 정책 강화
- DB 트랜잭션을 활용한 엑셀 업로드 일괄 저장 안정화

### 15. 마무리

`정산`은 단순히 영수증 글자를 읽는 서비스가 아니다. 흩어진 영수증과 거래내역을 하나의 장부 데이터로 통합하고, Supabase Postgres에 저장해 신고 준비에 필요한 요약을 만들어주는 프로젝트다.

이번 데이터베이스 수업 프로젝트에서는 이 서비스를 통해 현실의 문제를 데이터 모델로 바꾸는 과정을 보여주고자 했다. 특히 사용자, 사업장, 업로드 묶음, 파일 메타데이터, 장부 행, 신고 요약을 어떻게 분리하고 연결할 수 있는지, 그리고 AI가 만든 초안을 어떻게 검토 가능한 데이터로 관리할 수 있는지를 중점적으로 설계했다.

최종적으로 `정산`의 목표는 세금 신고를 완전히 자동화하는 것이 아니다. 사용자가 신고 전에 가장 오래 걸리는 자료 정리 시간을 줄이고, 믿고 확인할 수 있는 장부 데이터를 만드는 것이다.

## 발표용 짧은 요약

`정산`은 소상공인과 프리랜서를 위한 신고 준비 웹앱입니다. 영수증 사진과 거래내역 엑셀을 업로드하면 거래일, 거래처, 금액, 계정과목, 증빙유형을 장부 데이터로 정리하고, 사용자가 검토한 값을 Supabase Postgres에 저장합니다. DB 설계는 사용자, 사업장, 업로드 묶음, 파일 메타데이터, 장부 행, 신고 요약으로 분리했고, enum, FK, unique, check constraint, index, view, RLS를 적용해 데이터 무결성과 보안을 고려했습니다.
